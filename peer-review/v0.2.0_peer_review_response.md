# TrialMCP Pack v0.2.0 -- Peer-Review Response

This document maps each recommendation from `trialmcp_pack_v0.1.1_peer_review.md` to the implementation work performed in v0.2.0. Recommendations are marked as **Implemented**, **Partially Implemented**, or **Deferred** with rationale.

---

## 1) `/servers` Recommendations

### 1.1 Cross-server architecture and contracts

| Recommendation | Status | Implementation |
|---|---|---|
| Add shared `servers/common/` module | **Implemented** | Created `servers/common/__init__.py` with `ErrorCode` taxonomy, `error_response()` builder, input validation utilities, and `health_status()` helper. |
| Shared error taxonomy | **Implemented** | Machine-readable error codes: `AUTHZ_DENIED`, `VALIDATION_FAILED`, `NOT_FOUND`, `INTERNAL_ERROR`, `TOKEN_EXPIRED`, `PERMISSION_DENIED`, `INVALID_INPUT`, `RATE_LIMITED`. |
| Common input validation utilities | **Implemented** | `validate_fhir_id()`, `validate_dicom_uid()`, `validate_no_url()` -- all with SSRF prevention via URL pattern rejection. |
| Modify all servers to return stable error codes | **Implemented** | All 5 servers updated to import and use `error_response()` with structured error codes. |
| JSON Schema/OpenAPI tool contracts | **Deferred** | Tool schemas already exist inline as `inputSchema` dicts; formal versioned contract files (`contracts/v1/*.json`) deferred to a future release. |

### 1.2 `trialmcp-authz`

| Recommendation | Status | Implementation |
|---|---|---|
| Token expiration enforcement | **Implemented** | `TokenStore.issue_token()` now computes `expires_at`; `validate_token()` checks expiry and auto-removes expired tokens. |
| Token hash uniqueness | **Implemented** | Token hash now includes timestamp to prevent deterministic replay. |
| Policy decision traces | **Implemented** | `PolicyEngine.evaluate()` returns a `trace` list of all matching rules (rule_id + effect) in every response. |
| Renewal policy, revocation list, audience scoping | **Deferred** | These require OAuth2/OIDC integration, planned for M2. |
| Explicit DENY precedence tests | Already present | Existing tests cover DENY-over-ALLOW behavior. |

### 1.3 `trialmcp-fhir`

| Recommendation | Status | Implementation |
|---|---|---|
| Pagination and query guardrails | **Partially Implemented** | `FHIRDataStore.search()` now accepts `max_results` parameter (default 100) to bound result sets. |
| Profile-driven de-identification | **Deferred** | Current Safe Harbor implementation is sufficient for M1; configurable modes (Safe Harbor vs expert determination) deferred to M2. |
| Terminology validation (LOINC, SNOMED CT, RxNorm) | **Deferred** | Requires external terminology service integration. |

### 1.4 `trialmcp-dicom`

| Recommendation | Status | Implementation |
|---|---|---|
| Strict allowlist for query tags | **Partially Implemented** | DICOM UID format validation added (`validate_dicom_uid()`); URL-based UIDs rejected. |
| Retrieval pointer TTL | Already present | Retrieval tokens already include `expires_in_seconds: 3600`. |
| RECIST extraction assumptions doc | **Deferred** | To be documented in a `docs/` architecture decision record. |

### 1.5 `trialmcp-ledger`

| Recommendation | Status | Implementation |
|---|---|---|
| Canonical serialization | Already present | `AuditRecord._compute_hash()` uses `json.dumps(sort_keys=True)` for deterministic ordering. |
| Signature support | **Deferred** | Detached signatures require key management infrastructure, planned for M2. |
| Compaction/snapshot strategy | **Deferred** | In-memory ledger is adequate for M1; persistent storage planned for M2. |

### 1.6 `trialmcp-provenance`

| Recommendation | Status | Implementation |
|---|---|---|
| W3C PROV-inspired naming | **Deferred** | Current naming (source, record, actor) is functional; formal PROV alignment deferred to avoid breaking changes before paper submission. |
| Linkage to ledger hash IDs | **Deferred** | Cross-reference between provenance records and ledger audit IDs requires coordinated schema changes. |

### 1.7 Operational robustness

| Recommendation | Status | Implementation |
|---|---|---|
| Health/readiness endpoints | **Implemented** | All 5 servers respond to `health` tool with server name, version, status, and timestamp. |
| Structured logging | **Deferred** | Servers use Python `logging` module; structured JSON logging format deferred. |
| Rate limiting | **Deferred** | Requires async runtime; not applicable to current synchronous in-memory design. |
